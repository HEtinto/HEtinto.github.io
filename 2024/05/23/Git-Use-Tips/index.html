Git Use Tips

Git
Publish Date:   2024-05-23

Git切换分支

切换分支时提示切换分支失败，查询不到目标分支，此时应该使用下面的命令获取分支和标签信息：

git fetch
切换分支：
git checkout <分支名称>

Git上传代码

上传代码的流程：

git add .
git commit -m "信息"
git push
上传代码时遇到远程代码库与本地代码不一致的情况，可能是由于本地没有更新到最新的远程代码的原因，使用下面的流程：
首先保存本地修改后的代码保存到Git栈中：
git stash
然后将代码库更新到最新版本：
git pull
读取Git栈中最近保存的一次内容，恢复工作区中的内容：
git stash pop
最后使用命令上传即可：
git push
如果上面的方法失败了，也可以使用commit命令进行提交：
git commit -m "信息"
git push

代码提交注意事项

commit提交：一个事件，比如一个模块的修改，一个bug的修复，尽量只提交一次，不要一个事件提交多次，也不要两个事件提交一次。
force push：多人开发的分支，如果要强制提交，一般要跟一起开发的同事沟通好。

Git合并冲突

下载meld合并冲突解决工具后，安装到电脑并添加到系统环境变量，然后配置git文件发现meld：
打开配置文件：git config --system --edit

提交流程：
修改完成代码后，将代码合并到master分支：
git add .
git commit -m "infomation"
提交到当前分支：
git push
提交到master分支：
git push origin master
出现合并错误问题，使用meld比较文件差异：
git mergetool
解决好合并冲突后：
git commit -m "solve the merge error"
上传解决了合并冲突后的代码：
git push

分支命令

  1. git checkout -b branch_name : 创建分支并切换到创建好的分支；
  2. git rebase master : 更新master主线的内容到当前分支；
  3. git reset HEAD^ : 撤销上一次提交操作，保留本地变更——前提是没有推送.
  4. git merge branch_name : 本地分支合并branch_name分支.
  5. git branch -d branch_name : 删除分支branch_name，删除前确保该分支不被其他分支引用，删除后有30天时间可以撤销.
  6. git branch -D <branch_name> : 当使用 git branch -d删除失败时可以使用此命令再次尝试删除.
  7. git branch --contains <branch> : 查看分支是否与其它分支合并.
  8. git branch -m : 重命名分支.
  9. git branch -M : 如果你想重写此分支，可以使用此命令.
  10. git rm --cached : 忽略不想跟踪的文件.
  11. git add -f : 添加一个被忽略跟踪的文件.
  12. 编辑 .gitignore文件可以设置忽略跟踪的文件，比如一些编译后的二进制文件，隐私文件等.
  13. git status --ignored : 查看哪些文件被忽略了.
  14. 使用二进制属性宏将不易被识别为二进制文件的文件标记为二进制文件： （ps结尾的PostScript）*.ps binary、（xcode的编译配置文件.pbxproj）.
  15. git reset --soft <branch_version> : 当某一次提交中的文件有问题而需要修改时，可以使用此命令来撤销提交，这次撤销将会把日志回滚到特定的提交点并且不会删除被添加的文件，然后修改有问题的文件后再进行提交，这里的提交是commit，不是push，push是将本地推到远端。
  16. git rm --cached <file> : 撤销某个文件的特定操作。
  17. 临时提交，变更分支，回到原来的分支，恢复工作，删除提交：git cmmit -a -m “WIP”–>git checkout <other branch>–>git checkout - –>git reset –soft HEAD^–>git reset.
  18. git reset --hard HEAD^ : 强制撤销一个提交，会删除关于该提交的变更和缓存区信息，就好像该提交从未存在过，短时间内可通过reflog恢复。（特别重要的一点需要谨记：强制性重置操作是不可恢复地将暂存区和工作目录中的所有变更移除。用户无法撤销这部分操作！变更记录将会永远消失！）
  19. git reset --hard HEAD : 用于取消一个失败的合并操作（出现合并冲突但不想处理冲突时，否则也可以处理完冲突后成功合并）。
  20. git reset --hard HEAD ORIG_HEAD或git reset --hard HEAD HEAD@{1} : 用于移除一个成功执行的快进式拉取或变基操作以及其它移动分支首部的操作。
  21. git reset --keep HEAD^ : 回退分支并保留本地更改，暂存区中的内容将会删除，本地内容不变。git reset –soft命令暂存区中的内容不会被删除，本地内容也不会被删除。（暂存区中的内容为提交信息，文件变更信息）git reset –keep <revision>命令的工作原理是更新（工作目录下）我们回退的目标版本和分支首部引用版本之间有差异的文件内容。如果在分支首部和目标修订版本之间存在任何差异，并且本地文件存在未提交的变更记录，那么重置操作都会被终止。执行成功的情况和隐藏变更类似，强制性重置，然后取消隐藏变更。
  22. git checkout -b fixbug127-->gi reset --keep start 或 git stash-->git checkout -b fixbug127 start-->git stash pop : 将当前工作转移到一个新的更早的分支中。
  23. 使用–include-untracked选项暂存未跟踪的文件）以及暂存区的状态，系统保存该状态之后，通过运行git reset –hard HEAD命令，将会重置工作目录和索引，回退到最近一次提交的修订版本（为了匹配首部提交）。用户之后还可以随意地访问已经暂存的变更记录。
  24. git stash list : 查看暂存变更列表。
  25. 默认情况下，git stash pop命令会恢复最后一次暂存的变更记录，如果恢复成功，那么该记录将会从堆栈中删除。
  26. 用户可以在暂存列表中像使用diff命令那样通过git stash show -p命令查看暂存列表细节。
  27. git stash save "infomation" : 使用information信息来提示暂存的内容，git将使用该信息作为描述暂存信息。
  28. 当用户在目前工作的分支上执行git stash save后，因为该分支上新增了太多变更，以致于出现无法顺利执行git stash pop命令的现象，这主要是因为用户暂存变更后，新增了不少基于该修订的修订版本。如果用户希望在暂存变更之外再新建一个常规的提交，或者只是希望测试一下暂存的变更，那么可以使用git stash branch <分支名>命令。这会在用户保存变更的那个修订版本的基础上新建一个分支并切换到该分支，恢复用户之前保存的变更，然后将上述暂存的变更在暂存列表中删除。
  29. 用户可以使用git stash命令保留索引的状态，然后通过使用–keep-index选项将工作区重置到暂存状态。
  30. 用户还可以使用git stash –patch命令在将变更暂存之后，指定工作区的表现形式。
  31. git grep -e ';;':在当前工作目录下向下搜索所有被跟踪的文件，还可以限定搜索的文件类型，比如搜索范围限制在c源代码文件中：git grep -e ';;' -- "*.c，还可以使用git grep --cached来搜索暂存区的内容。
  32. 撤销添加未跟踪文件到索引的操作，或者从暂存区移除一个被跟踪文件，以便在下一个提交中移除该文件，那么可以在保持工作目录状态的同时，使用git rm–cached <file>命令达到此目的。
  33. 可以使用git reset HEAD – <file>命令将暂存的文件内容重置到以前的修订版本。
  34. 可以使用git checkout – <file>命令从索引中将工作区的状态恢复到以前的版本。如果用户暂存的某个状态非常混乱，也可以使用git checkout HEAD – <file>命令，将该状态恢复到最近一次提交的状态。
  35. 用户希望使用以前的某个修订版本文件替换当前工作区的src/rand.c文件，那么可以使用git checkout HEAD^– src/rand.c命令（或者用git show HEAD^:src/rand.c命令将输出结果重定向到某个文件）。为了将next分支上的版本添加到暂存区，用户可以执行git reset next – src/rand.c命令。
  36. git clean --dry-run : Git系统会从当前目录开始递归清除所有未被跟踪的文件。
  37. git merge --no-ff 目标分支:指定提交为非快进式提交。（快进式提交是指：假设有A和B两个分支都有公共父节点master，其中A分支与master之间还有两个提交m1，f1和f2，而B分支与master分支之间有一个提交m1，此时将B分支与A分支合并，会直接将master分支指向A分支，而不是产生一个合并提交，即直接将B分支快进到A分支。）
  38. git push --set-upstream origin baran_name:建立分支后要的第一次推送——将分支与远程关联然后推送。

协作开发

1.获取远程版本库别名

为了获取目前配置的远程版本库的信息，可以执行git remote命令，它会列出用户拥有的所有远程版本库的别名。使用git remote -v选项可以显示远程库的url。

2.新建一个远程版本库

为了新建一个Git的远程版本库，并给它添加一个别名以便记忆，可以执行git remote add <shortname> <URL>命令进行提交。在执行上述命令时，系统并不会拉取远程版本库的信息到本地，通过添加-f参数可以实现拉取远程版本库信息。用户可以使用-t <branch>选项在远程版本库中指定特定分支，还可以使用-m <branch>选项为远程版本库指定默认分支（以及相关的远程分支名称），也可以使用--mirror=push或者--mirror=fetch选项配置远程版本库镜像方便协作。

3.远程版本库名称和url修改

执行git remote rename <old> <new>命令后，不仅会修改remote.<old>中的名字，而且还会修改远程跟踪分支和相关的refspec，即引用日志（详情可以参考帮助手册中core.logAllRef Updates配置变量的介绍）以及它们对应的分支配置。
用户可以使用git remote set-url命令为远程版本库添加或者替换URL地址，不过也可以直接编辑相关的配置文件进行修改。
修改URL常见的一种情况是修改远程版本库的跟踪分支列表（拉取内容的列表）：用户可以使用git remote set-branches命令（通过一款当前最新的高效Git软件实现）或者直接手动编辑配置文件实现该目的。

4.智能传输

Git智能感知服务器会使用git upload-pack命令下载更新，使用git receive-pack命令推送变更。

5.推送变更到公共版本库

在中心式工作流中，用户只需要将本地的变更推送到中心服务器即可，但是在这个过程中其它项目成员可能已经推送了新的变化，所以需要先使用pull命令同步新的变化，然后再推送自己的更新。
在使用个人公共版本库的工作流中，用户需要向合作开发者、维护者或者集成经理发送有新的变更需要推送的通知。git request-pull命令可以帮助用户完成此目的。
推送到master节点的步骤：

  1. git checkout master;
  2. git pull;
  3. git merge feature-foo;
  4. git pull origin master.

一种替代性的解决方案是对远程跟踪分支上的主题分支进行变基要好过合并它们。变基之后，主题分支将变成远程版本库中master分支一个祖先，因此推送到master分支会很方便：

用户可以首先使用git merge –no-commit命令让Git系统不要执行自动合并来检查它，或者用户发现合并记录中有问题时，可以先检查相应的合并记录，然后使用git commit –amend命令对它们进行纠正。

6.创建孤儿分支

使用git checkout –orphan命令创建一个孤儿分支时，从技术上来说并没有新建一个分支引用。实际的行为是将HEAD标识符引用指向了一个未出生的分支，即在新的孤儿分支上添加第一个提交对象之后创建了上述引用。

8.合并冲突的产生和解决

三路合并规则：
如果一方修改变更了某个文件，选择修改版的文件。如果双方拥有相同的变更，选择修改过的版本。如果一方包含与另一方不一样的变更，那么在内容层面存在合并冲突。三路文件合并会检查变更是否涉及文件内的不同部分（有多少行发生了修改，这些内容变更之间都是经过差异上下文标记进行良好的区隔）。如果这些变更出现在了文件中不同的部分，Git会自动合并它们（并告知用户哪些文件被自动合并了）。
当Git系统无法自动处理一个合并时（或者用户在执行git merge命令时指定了–no-commit参数选项），它将不会创建一个合并提交。Git会终止合并操作，等待用户自行解决合并冲突。

如果在执行合并操作之前的工作区是干净的，那么可以使用命令git merge --abort重置状态到执行合并操作之前，所以在执行一个合并操作之前，最好将变更暂存。

在合并前如果有不需要进行合并的工作，那么可以将其添加到暂存区，然后使用git stash命令保存到栈中，然后执行合并，合并执行完后，使用git stash pop弹出栈顶保存的工作区内容，恢复工作区。

  1. 从目标地址获取代码：git clone http://hci.macrosan.com/hci/hci_python.git
  2. 切换到对应分支：git checkout feat_gran_pool
  3. 更新代码：git pull
  4. 编码操作：…
  5. 将所有更改的文件添加到暂存区：git add .
  6. 提交更改：git commit -m "pool和osd流程修改"
  7. 发现问题：没有添加报错码
  8. 解决方法：添加错误码，新增一次提交
  9. 简化提交信息：将前两次提交合并为一次
  10. 解决方式：保留本地工作区内容并回退分支
  11. 回退分支并保留本地更改（branch_version为需要回退的目标版本号）：git reset --soft branch_version
  12. 添加提交信息：git commit -m "pool和osd流程修改"
  13. 推送代码：git push --force
